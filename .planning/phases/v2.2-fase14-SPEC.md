# SPEC: Fase 14 — Fluxo Administrativo e Avaliação de Viabilidade

## 1. Definição do Problema
O engenheiro precisa delegar a confecção de documentos acessórios (certidões) ao setor administrativo. Para isso, o Parecer deve ser instrutivo, contendo os dados validados e os textos padronizados, evitando que o administrativo precise re-analisar o processo.

## 2. Requisitos Funcionais

### RF-14.1: Avaliação de Viabilidade (Eligibility)
O sistema deve aplicar a seguinte lógica na Fase 5:
- **Certidão de Localização**: 
    - *Condição*: Presença de Matrícula + Inscrição Municipal + Zona.
    - *Bloqueio*: Se Matrícula ausente, marcar como "Inviável — Aguardar Registro".
- **Certidão de Nome de Rua**: 
    - *Condição*: Nome extraído diverge do oficial (MCP) + Existência de Decreto.
    - *Bloqueio*: Se Decreto não localizado, marcar como "Inviável — Consultar Arquivo de Decretos".
- **Certidão de Confrontação**: 
    - *Condição*: Memorial descritivo legível.
    - *Bloqueio*: Se memorial ilegível, marcar como "Inviável — Exigir novo memorial".

### RF-14.2: Bloco de Dados Administrativos
Para cada certidão viável, o sistema deve gerar um objeto no JSON:
```json
{
  "tipo": "certidao_localizacao",
  "viabilidade": "apto",
  "texto_padrao": "Certificamos para os devidos fins que o imóvel...",
  "variaveis": {
    "logradouro": "Rua X",
    "bairro": "Y",
    "matricula": "123"
  }
}
```

### RF-14.3: Interface SIA (Fase 5)
A Fase 5 do SIA será expandida para mostrar:
- Lista de certidões propostas.
- Status de Viabilidade (✅ Apto / ❌ Inviável).
- Motivo da inviabilidade (se houver).
- Dados que serão enviados para o administrativo.

## 3. Padrões de Texto (Knowledge)
Deverá ser criado um novo arquivo `Sistema/base_conhecimento/padroes_certidoes.md` contendo os templates que a IA deve usar para preencher o campo `texto_padrao`.
